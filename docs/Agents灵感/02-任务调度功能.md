# 02 - 任务调度功能

## 1. 概述

任务调度是 Agent 系统的核心引擎，负责管理任务的创建、排队、分配和执行编排。它将用户的需求转化为 Agent 可执行的指令，并按照优先级、依赖关系和资源约束进行智能调度。

## 2. 任务类型扩展

### 2.1 现有 Todo 与 Agent 任务的关系

```
Todo 项
├── 普通 Todo（保持现有功能不变）
│   ├── 标题、描述、优先级、子任务...
│   └── 手动完成/取消
│
└── Agent Todo（新增类型）
    ├── 继承普通 Todo 的所有属性
    ├── + 目标项目路径 (project_path)
    ├── + Prompt 内容 (prompt)
    ├── + 指定 Agent (agent_id)
    ├── + 执行策略 (strategy)
    ├── + 执行状态 (agent_status)
    └── Agent 执行完成 = Todo 进入待审核状态
```

### 2.2 Agent 任务属性

| 属性 | 类型 | 必填 | 说明 |
|------|------|------|------|
| project_path | string | 是 | 目标项目的本地路径 |
| prompt | string | 是 | 发送给 Agent 的指令 |
| agent_id | number | 否 | 指定使用的 Agent（空则自动分配） |
| strategy | enum | 否 | 执行策略：auto / manual / scheduled |
| branch_name | string | 否 | Agent 工作的 Git 分支名 |
| context_files | string[] | 否 | 需要 Agent 重点关注的文件列表 |
| max_retries | number | 否 | 最大重试次数（默认 0） |
| priority_score | number | 否 | 调度优先分数（0-100） |

## 3. Prompt 模板系统

### 3.1 内置模板

| 模板名称 | 适用场景 | 变量 |
|----------|---------|------|
| 功能开发 | 新增功能 | {feature_desc}, {tech_stack}, {file_scope} |
| Bug 修复 | 修复问题 | {bug_desc}, {reproduce_steps}, {expected_behavior} |
| 代码重构 | 优化代码 | {refactor_scope}, {refactor_goal}, {constraints} |
| 单元测试 | 编写测试 | {target_file}, {test_framework}, {coverage_goal} |
| Code Review | 代码审查 | {review_scope}, {review_focus}, {coding_standards} |
| 文档生成 | 编写文档 | {doc_scope}, {doc_type}, {audience} |
| 依赖升级 | 升级依赖 | {package_name}, {target_version}, {breaking_changes} |

### 3.2 模板结构

```
PromptTemplate {
    id: string
    name: string              // 模板名称
    category: string          // 分类
    description: string       // 模板说明
    template: string          // 模板内容（支持 {{variable}} 占位符）
    variables: Variable[]     // 变量定义
    recommended_agent: string // 推荐使用的 Agent 类型
    is_builtin: boolean       // 是否内置模板
}

Variable {
    name: string
    label: string             // 显示标签
    type: text | textarea | select | file_picker
    required: boolean
    default_value?: string
    options?: string[]        // type=select 时的选项
}
```

### 3.3 自定义模板

- 用户可基于内置模板修改创建自定义模板
- 支持导入/导出模板（JSON 格式）
- 模板内可引用项目级上下文（如 CLAUDE.md 的内容）

## 4. 任务依赖链

### 4.1 依赖类型

| 类型 | 说明 | 示例 |
|------|------|------|
| finish-to-start | A 完成后 B 才能开始 | 先写代码，再写测试 |
| finish-to-finish | A 完成后 B 才能标记完成 | 代码和文档同时开发，但文档需等代码定稿 |
| start-to-start | A 开始后 B 才能开始 | 并行任务但有启动顺序 |

### 4.2 依赖关系模型

```
TaskDependency {
    id: number
    task_id: number         // 当前任务
    depends_on_id: number   // 依赖的任务
    dependency_type: string // finish-to-start | finish-to-finish | start-to-start
}
```

### 4.3 DAG 可视化

- 在任务详情页展示依赖关系图
- 使用简单的树状或流程图展示
- 高亮当前可执行的任务节点
- 标记阻塞路径

### 4.4 循环依赖检测

- 创建依赖时自动检测是否形成环
- 拒绝形成循环依赖的操作
- 提供友好的错误提示

## 5. 触发机制

### 5.1 手动触发

- 用户在任务详情页点击 "执行" 按钮
- 任务立即进入调度队列

### 5.2 定时触发

| 方式 | 说明 |
|------|------|
| 一次性定时 | 设定具体的执行时间 |
| Cron 表达式 | 如 `0 9 * * 1` 表示每周一早上 9 点 |
| 周期重复 | 每小时 / 每天 / 每周 / 每月 |

### 5.3 条件触发（进阶）

| 触发条件 | 说明 |
|----------|------|
| Git Push | 目标仓库有新 Push 时触发 |
| 文件变更 | 监视指定文件/目录的变更 |
| 上游任务完成 | 依赖任务完成后自动触发 |
| 手动审批通过 | 上一任务审批通过后触发下一步 |

### 5.4 定时任务使用场景

- 每日代码质量检查
- 每周依赖安全审计
- 定期生成项目文档
- 定时同步/备份操作

## 6. 并发控制

### 6.1 全局控制

| 参数 | 说明 | 默认值 |
|------|------|--------|
| max_global_concurrent | 系统最大并行任务数 | 3 |
| max_queue_size | 最大排队长度 | 50 |
| queue_timeout | 排队超时时间 | 30 分钟 |

### 6.2 Agent 级别控制

- 每个 Agent 有独立的 `max_concurrent` 设置
- 同一 Agent 的并发任务不超过其限制
- 不同 Agent 的任务可以并行

### 6.3 项目级别控制

- 同一项目同时只能有 N 个 Agent 任务运行
- 避免多个 Agent 同时修改同一项目导致冲突
- 默认建议同一项目最大并发为 1

### 6.4 资源锁

```
当任务 A 正在修改 project-x 时：
  → 任务 B（同一项目）自动排队等待
  → 任务 C（不同项目）可以并行执行
```

## 7. 优先级队列

### 7.1 优先级计算

```
最终优先分 = 基础优先级 × 3 + 等待时间加成 + 截止日期加成 + 用户指定加成
```

| 因素 | 计算方式 |
|------|---------|
| 基础优先级 | 高=30, 中=20, 低=10 |
| 等待时间加成 | 每等待 10 分钟 +1 分（防饥饿） |
| 截止日期加成 | 截止日期临近时额外加分 |
| 用户指定加成 | 用户可手动设定 0-100 的额外加分 |

### 7.2 队列行为

- 新任务进入队列时按优先分排序
- 有空闲 Agent 资源时从队首取任务
- 支持手动调整队列顺序（拖拽排序）
- 支持紧急任务插队（需确认）

## 8. 调度状态流转

```
创建 → 待调度 → 排队中 → 执行中 → 执行完成
                                      ↓
                              ┌───────┴───────┐
                              ↓               ↓
                          待审核          执行失败
                              ↓               ↓
                      ┌───────┴───────┐   重试/放弃
                      ↓               ↓
                  审核通过        审核拒绝
                      ↓               ↓
                  已完成          返回待调度
```

### 状态说明

| 状态 | 说明 |
|------|------|
| draft | 草稿，尚未提交 |
| pending | 待调度，等待进入队列 |
| queued | 在队列中排队等待 |
| running | Agent 正在执行 |
| completed | 执行完成，等待审核 |
| failed | 执行失败 |
| reviewing | 人工审核中 |
| approved | 审核通过 |
| rejected | 审核拒绝，可重新调度 |
| cancelled | 已取消 |

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PRIMARY KEY | 自增主键 |
| todo_id | INTEGER | 关联的 Todo 项 ID（可空） |

| strategy | TEXT DEFAULT 'manual' | 执行策略 |
| status | TEXT DEFAULT 'draft' | 当前状态 |
| branch_name | TEXT | Git 工作分支 |
| context_files | TEXT | JSON 格式的关注文件列表 |
| priority_score | INTEGER DEFAULT 0 | 调度优先分 |
| max_retries | INTEGER DEFAULT 0 | 最大重试次数 |
| retry_count | INTEGER DEFAULT 0 | 已重试次数 |
| scheduled_at | TEXT | 计划执行时间 |

| 字段 | 类型 | 说明 |

### 9.3 新增表：prompt_templates

| 字段 | 类型 | 说明 |

| id | TEXT PRIMARY KEY | 模板 ID |

| category | TEXT | 分类 |
| description | TEXT | 说明 |
