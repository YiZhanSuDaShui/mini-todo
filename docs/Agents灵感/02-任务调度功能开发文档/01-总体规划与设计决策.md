# 任务调度功能 - 总体规划与设计决策

## 1. 开发目标

基于 [02-任务调度功能.md](../02-任务调度功能.md) 灵感文档，实现任务调度系统，使系统具备：

1. 基于现有 Todo + Subtask 体系的 Agent 任务调度
2. 任务状态流转（创建 → 排队 → 执行 → 完成/失败）
3. 执行超时自动终止
4. 优先级队列与并发控制
5. 任务依赖链管理
6. 并发冲突自动解决策略（Git merge/rebase）
7. 简单 Prompt 模板（基于现有 prompt 模式）
8. 定时任务管理界面

## 2. 关键设计决策

基于灵感文档第 10 节的讨论结果，明确以下设计方向：

| # | 问题 | 决策 | 对实现的影响 |
|---|------|------|-------------|
| 1 | 任务与 Todo 关系 | **基于现有 Todo 和 Subtask 关系** | 不新建独立的 `agent_tasks` 表，而是扩展现有的 `todos`/`subtasks` 表增加调度相关字段 |
| 2 | Prompt 模板继承 | **基于现有的 prompt** | 简化模板系统，利用子任务的标题+内容作为 prompt 基础，模板仅提供预设模式，不支持继承 |
| 3 | 定时任务管理 | **需要独立管理界面** | 新增定时任务列表页面/面板，支持 Cron 可视化配置 |
| 4 | 并发冲突解决 | **需要优秀的自动解决策略** | 实现 Git auto-merge/rebase 策略，同项目互斥锁，冲突时自动回退 |
| 5 | 执行超时 | **自动终止，设为"任务失败"** | 超时后 kill 进程，状态标记 `failed`，记录超时原因 |
| 6 | 通知方式 | **目前不需要** | 暂不实现任务失败通知，后续再考虑 |

## 3. 与现有架构的关系

### 3.1 现有架构回顾

```
当前体系：
  Todo （待办事项）
    ├── 基本信息：title, description, color, quadrant, priority...
    ├── Agent 配置：agent_id, agent_project_path
    └── Subtask （子任务）
          ├── 基本信息：title, content, completed
          └── Agent 执行：通过 SubtaskEditorView 手动触发
                └── agent_executions 表记录执行结果
```

### 3.2 调度系统的扩展方向

```
扩展后体系：
  Todo （待办事项）
    ├── 基本信息（不变）
    ├── Agent 配置（不变）
    ├── + 调度策略：strategy (manual/auto/scheduled)
    ├── + 定时配置：scheduled_at, cron_expression
    └── Subtask （子任务 = Agent 任务单元）
          ├── 基本信息（不变）
          ├── + 调度状态：schedule_status (draft/pending/queued/running/...)
          ├── + 优先级分数：priority_score
          ├── + 最大重试次数：max_retries
          ├── + 已重试次数：retry_count
          ├── + 依赖关系：通过 task_dependencies 表
          └── Agent 执行：
                ├── 手动触发（现有功能）
                ├── 自动调度触发（新增）
                └── 定时触发（新增）
```

**核心思路：子任务（Subtask）作为 Agent 任务的最小调度单元。**

- 每个子任务可以独立被调度执行
- 父 Todo 提供 Agent 配置（agent_id、project_path）和调度策略
- 子任务的 title + content 构成发送给 Agent 的 prompt

## 5. 分阶段实现计划

### Phase 1：核心调度基础（预计 3-5 天）

**目标**：为现有系统增加调度能力的基础设施。

| 任务 | 说明 | 涉及文件 |
|------|------|---------|
| 1.1 数据库迁移 | 扩展 `subtasks` 表增加调度字段；创建 `task_dependencies` 表 | `migrations.rs`, `models.rs` |
| 1.2 超时自动终止 | 在 `runner.rs` 的 `run_process` 中加入超时检测，超时后 kill 进程并标记 failed | `runner.rs` |
| 1.3 任务状态流转 | 实现 subtask 的调度状态机（draft → pending → queued → running → completed/failed） | `scheduler_db.rs`, `models.rs` |
| 1.4 基本队列 | 实现内存优先级队列，支持入队/出队/查看 | `services/scheduler/queue.rs` |

### Phase 2：调度引擎（预计 5-7 天）

**目标**：实现完整的调度引擎，支持自动执行。

| 任务 | 说明 | 涉及文件 |
|------|------|---------|
| 2.1 优先级计算 | 实现优先级评分算法（基础优先级 + 等待时间 + 截止日期） | `queue.rs` |
| 2.2 并发控制 | 全局并发限制、同项目互斥锁 | `executor.rs` |
| 2.3 调度循环 | 后台 tokio 任务，持续从队列中取任务并分配执行 | `executor.rs` |
| 2.4 任务依赖链 | DAG 模型，循环检测，依赖完成自动触发下游任务 | `dependency.rs`, `dependency_db.rs` |
| 2.5 Git 冲突解决 | 同项目并发时的自动 merge/rebase 策略 | `conflict.rs` |
| 2.6 重试机制 | 失败任务自动重试（根据 max_retries 配置） | `executor.rs` |

### Phase 3：Prompt 模板（预计 2-3 天）

**目标**：提供预设模板简化 prompt 编写。

| 任务 | 说明 | 涉及文件 |
|------|------|---------|
| 3.1 模板数据库 | 创建 `prompt_templates` 表，内置常用模板 | `migrations.rs`, `models.rs` |
| 3.2 模板 CRUD | 模板的增删改查 API | `scheduler_cmd.rs`, `scheduler_db.rs` |
| 3.3 模板选择 UI | 在子任务编辑页面增加模板选择器 | `SubtaskEditorView.vue` |
| 3.4 变量替换 | 模板中 `{{variable}}` 占位符的解析和替换 | `executor.rs` |

### Phase 4：定时任务管理（预计 3-5 天）

**目标**：独立的定时任务管理界面和 Cron 调度能力。

| 任务 | 说明 | 涉及文件 |

| 4.1 Cron 解析器 | 实现 Cron 表达式解析和下次执行时间计算 | `cron.rs` |
| 4.2 定时调度器 | 后台 tokio 任务，检查定时任务并触发 | `executor.rs` |

## 6. 技术选型

| 需求 | 方案 | 说明 |
|------|------|------|
| 优先级队列 | `BinaryHeap` + 自定义 `Ord` | Rust 标准库，内存队列，重启后从数据库恢复 |
| 并发控制 | `tokio::sync::Semaphore` + `RwLock<HashMap>` | 全局信号量控制并发数，HashMap 记录项目锁 |
| Cron 解析 | `cron` crate | Rust 的 Cron 表达式解析库 |
| 后台调度 | `tokio::spawn` + `tokio::time::interval` | 定期检查队列和定时任务 |
| DAG 检测 | DFS 拓扑排序 | 创建依赖时实时检测环 |
| Git 操作 | `std::process::Command` 调用 git | 与现有 worktree 方案一致 |
| 前端依赖图 | 简单 Canvas/SVG 或 `vue-flow` | 任务依赖关系可视化 |

## 7. 风险与注意事项

### 7.1 数据一致性
- 调度状态存于数据库，内存队列仅作缓存
- 应用重启时从数据库重建队列
- 使用事务确保状态更新的原子性

### 7.2 超时处理
- 默认超时时间 600 秒（10 分钟），可配置
- 超时后先发 SIGTERM，等待 10 秒后强制 SIGKILL
- Windows 下使用 `taskkill` 进行进程终止

### 7.3 并发冲突解决策略

```
当同一项目有多个任务并发时：

策略 1：互斥锁（默认）
  → 同一项目同时只允许一个 Agent 任务运行
  → 其他任务自动排队等待

策略 2：分支隔离
  → 每个任务在独立 Git 分支上工作
  → 完成后尝试自动 merge 到主分支
  → merge 冲突时标记任务为 "需要手动解决"

策略 3：Worktree 隔离（进阶）
  → 每个任务使用独立的 Git Worktree
  → 并行执行，完成后依次合并
  → 合并失败时自动 rebase 重试一次
```

### 7.4 与现有功能的兼容性
- 不改变现有手动执行流程，调度功能为**增量扩展**
- 现有的 `agent_executions` 表继续使用，调度器通过它记录执行结果
- `SubtaskEditorView` 的手动执行功能保持不变
- 新增调度相关的 UI 与现有 UI 独立

### 7.5 性能考虑
- 优先级队列操作 O(log n)
- 定时检查间隔建议 5-10 秒
- 同时运行的 Agent 任务建议不超过 3 个（受本机资源限制）
