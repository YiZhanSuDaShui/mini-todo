# 任务调度功能 - 数据库设计

## 1. 设计原则

- **扩展现有表**：调度相关字段扩展到 `subtasks` 表，而非新建独立的 `agent_tasks` 表
- **最小侵入**：新增字段全部有默认值，不影响现有功能
- **状态持久化**：调度队列状态持久化到数据库，支持应用重启恢复

## 2. 表变更详情

### 2.1 扩展 subtasks 表

新增以下字段：

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| schedule_status | TEXT | 'none' | 调度状态：none/draft/pending/queued/running/completed/failed/reviewing/approved/rejected/cancelled |
| priority_score | INTEGER | 0 | 调度优先级分数（0-100） |
| max_retries | INTEGER | 0 | 最大重试次数 |
| retry_count | INTEGER | 0 | 已重试次数 |
| timeout_secs | INTEGER | 600 | 执行超时秒数，默认 10 分钟 |
| scheduled_at | TEXT | NULL | 一次性定时执行时间（ISO 8601） |
| last_scheduled_run | TEXT | NULL | 上次定时执行时间 |
| schedule_error | TEXT | NULL | 最近一次调度错误信息 |

**迁移 SQL：**

```sql
-- migration_v13
ALTER TABLE subtasks ADD COLUMN schedule_status TEXT NOT NULL DEFAULT 'none';
ALTER TABLE subtasks ADD COLUMN priority_score INTEGER NOT NULL DEFAULT 0;
ALTER TABLE subtasks ADD COLUMN max_retries INTEGER NOT NULL DEFAULT 0;
ALTER TABLE subtasks ADD COLUMN retry_count INTEGER NOT NULL DEFAULT 0;
ALTER TABLE subtasks ADD COLUMN timeout_secs INTEGER NOT NULL DEFAULT 600;
ALTER TABLE subtasks ADD COLUMN scheduled_at TEXT;
ALTER TABLE subtasks ADD COLUMN last_scheduled_run TEXT;
ALTER TABLE subtasks ADD COLUMN schedule_error TEXT;

CREATE INDEX IF NOT EXISTS idx_subtasks_schedule_status ON subtasks(schedule_status);
```

**字段说明：**

- `schedule_status = 'none'` 表示该子任务不参与调度（传统手动模式）
- 只有 `schedule_status` 不为 `none` 的子任务才进入调度系统
- 手动通过 SubtaskEditorView 执行的子任务不受调度状态影响

### 2.2 扩展 todos 表

新增以下字段：

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| schedule_strategy | TEXT | 'manual' | 调度策略：manual（手动触发）/ auto（自动调度）/ scheduled（定时执行） |
| cron_expression | TEXT | NULL | Cron 表达式（仅 strategy=scheduled 时有效） |
| schedule_enabled | INTEGER | 0 | 定时调度是否启用 |

**迁移 SQL：**

```sql
-- migration_v14
ALTER TABLE todos ADD COLUMN schedule_strategy TEXT NOT NULL DEFAULT 'manual';
ALTER TABLE todos ADD COLUMN cron_expression TEXT;
ALTER TABLE todos ADD COLUMN schedule_enabled INTEGER NOT NULL DEFAULT 0;
```

**字段说明：**

- `schedule_strategy = 'manual'`：保持现有行为，子任务需手动触发执行
- `schedule_strategy = 'auto'`：子任务被标记为 `pending` 后自动进入调度队列
- `schedule_strategy = 'scheduled'`：按 `cron_expression` 定时触发子任务执行

### 2.3 新建 task_dependencies 表

管理子任务之间的依赖关系。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PRIMARY KEY | 自增主键 |
| subtask_id | INTEGER NOT NULL | 当前子任务 ID |
| depends_on_id | INTEGER NOT NULL | 依赖的子任务 ID |
| dependency_type | TEXT | 依赖类型：finish-to-start / finish-to-finish / start-to-start |
| created_at | TEXT | 创建时间 |

**迁移 SQL：**

```sql
-- migration_v15
CREATE TABLE IF NOT EXISTS task_dependencies (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    subtask_id      INTEGER NOT NULL,
    depends_on_id   INTEGER NOT NULL,
    dependency_type TEXT    NOT NULL DEFAULT 'finish-to-start',
    created_at      TEXT    NOT NULL DEFAULT (datetime('now', 'localtime')),
    FOREIGN KEY (subtask_id) REFERENCES subtasks(id) ON DELETE CASCADE,
    FOREIGN KEY (depends_on_id) REFERENCES subtasks(id) ON DELETE CASCADE,
    UNIQUE(subtask_id, depends_on_id)
);

CREATE INDEX IF NOT EXISTS idx_task_deps_subtask ON task_dependencies(subtask_id);
CREATE INDEX IF NOT EXISTS idx_task_deps_depends ON task_dependencies(depends_on_id);
```

**约束说明：**
- `UNIQUE(subtask_id, depends_on_id)` 防止重复依赖
- `ON DELETE CASCADE` 删除子任务时自动清除相关依赖
- 应用层需实现循环依赖检测（DFS）

### 2.5 新建 schedule_history 表（可选）

记录定时任务的触发历史。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PRIMARY KEY | 自增主键 |
| todo_id | INTEGER NOT NULL | 关联的 Todo ID |
| subtask_id | INTEGER | 关联的子任务 ID（可空） |
| trigger_type | TEXT | 触发类型：manual / cron / dependency / condition |
| trigger_time | TEXT | 触发时间 |
| execution_id | INTEGER | 关联的 agent_executions ID |
| status | TEXT | 结果状态 |
| error | TEXT | 错误信息 |

```sql
-- migration_v17 (可选)
CREATE TABLE IF NOT EXISTS schedule_history (
    id              INTEGER PRIMARY KEY AUTOINCREMENT,
    todo_id         INTEGER NOT NULL,
    subtask_id      INTEGER,
    trigger_type    TEXT    NOT NULL DEFAULT 'manual',
    trigger_time    TEXT    NOT NULL DEFAULT (datetime('now', 'localtime')),
    execution_id    INTEGER,
    status          TEXT    NOT NULL DEFAULT 'triggered',
    error           TEXT,
    FOREIGN KEY (todo_id) REFERENCES todos(id) ON DELETE CASCADE,
    FOREIGN KEY (subtask_id) REFERENCES subtasks(id) ON DELETE SET NULL,
    FOREIGN KEY (execution_id) REFERENCES agent_executions(id) ON DELETE SET NULL
);

CREATE INDEX IF NOT EXISTS idx_schedule_history_todo ON schedule_history(todo_id);
```

## 3. 状态流转图

### 3.1 子任务调度状态（schedule_status）

```
none ────────────────────────────────────────────────────────────
 │                    （不参与调度，传统手动模式）
 │
 ▼ 用户启用调度
draft ──→ pending ──→ queued ──→ running ──→ completed
              │          │          │            │
              │          │          │            ▼
              │          │          │        reviewing ──→ approved ──→ (subtask.completed = true)
              │          │          │            │
              │          │          │            ▼
              │          │          │        rejected ──→ pending (重新调度)
              │          │          │
              │          │          ▼
              │          │       failed ──→ pending (自动重试, retry_count < max_retries)
              │          │          │
              │          │          └──→ cancelled (超过重试次数或手动取消)
              │          │
              └──────────┴──→ cancelled (用户手动取消)
```

### 3.2 状态转换触发条件

| 从 → 到 | 触发条件 | 执行操作 |
|---------|---------|---------|
| none → draft | 用户对子任务启用调度 | 更新 schedule_status |
| draft → pending | 用户提交调度 / 自动模式 | 检查依赖是否满足 |
| pending → queued | 依赖已满足，进入队列 | 计算优先级分数，入队 |
| queued → running | 调度器取出任务开始执行 | 调用 AgentManager |
| running → completed | Agent 执行成功 | 记录结果到 agent_executions |
| running → failed | Agent 执行失败或超时 | 记录错误，判断是否重试 |
| failed → pending | retry_count < max_retries | retry_count++ |
| failed → cancelled | 超过重试次数 | 终态 |
| completed → reviewing | 配置了人工审核 | 等待用户审核 |
| reviewing → approved | 用户确认通过 | 标记子任务 completed |
| reviewing → rejected | 用户拒绝 | 回到 pending 重新执行 |

## 5. 导入导出兼容性

根据项目的 CLAUDE.md 规范，数据库结构变更时需同步检查导入导出功能：

- `subtasks` 新增的调度字段需要在 `ExportData` 中体现
- 新字段都有默认值，旧版数据导入不会出错（`#[serde(default)]`）
- `task_dependencies` 表是否需要导出：**需要**，因为它关联了子任务间的逻辑关系
- `prompt_templates` 中 `is_builtin = 1` 的模板不需要导出（导入时自动由迁移创建）
- `schedule_history` 不需要导出（历史数据与当前机器绑定）
- 导出版本号需要递增

## 6. 常用查询

### 6.1 查询可调度的子任务（依赖已满足）

```sql
SELECT s.* FROM subtasks s
WHERE s.schedule_status = 'pending'
  AND NOT EXISTS (
    SELECT 1 FROM task_dependencies td
    JOIN subtasks dep ON dep.id = td.depends_on_id
    WHERE td.subtask_id = s.id
      AND td.dependency_type = 'finish-to-start'
      AND dep.schedule_status NOT IN ('completed', 'approved')
  )
ORDER BY s.priority_score DESC;
```

### 6.2 查询某个子任务的所有依赖

```sql
SELECT td.*, s.title as depends_on_title, s.schedule_status as depends_on_status
FROM task_dependencies td
JOIN subtasks s ON s.id = td.depends_on_id
WHERE td.subtask_id = ?;
```

### 6.3 循环依赖检测（应用层实现）

```rust
fn has_circular_dependency(
    conn: &Connection,
    subtask_id: i64,
    depends_on_id: i64,
) -> Result<bool> {
    // DFS: 从 depends_on_id 出发，看是否能回到 subtask_id
    let mut visited = HashSet::new();
    let mut stack = vec![depends_on_id];

    while let Some(current) = stack.pop() {
        if current == subtask_id {
            return Ok(true); // 形成环
        }
        if visited.contains(&current) {
            continue;
        }
        visited.insert(current);

        // 查找 current 依赖的所有任务
        let deps = get_dependencies(conn, current)?;
        for dep in deps {
            stack.push(dep.depends_on_id);
        }
    }
    Ok(false)
}
```
