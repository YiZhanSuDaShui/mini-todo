# 04 - 结果管理与审核

## 1. 概述

Agent 执行完成后，其产出的代码变更需要经过人工审核才能正式生效。结果管理与审核模块提供代码 Diff 预览、审批流程、Git 集成、结果评分等能力，确保 AI 生成的代码质量可控。

## 2. 代码变更收集

### 2.1 变更检测方式

| 方式 | 说明 | 适用场景 |
|------|------|---------|
| Git Diff | Agent 执行前后的 `git diff` | Agent 在 Git 仓库中工作 |
| 文件快照对比 | 执行前后的文件哈希对比 | 非 Git 项目 |
| Agent 输出解析 | 从 Agent 输出中提取变更信息 | Agent 提供结构化输出时 |

### 2.2 变更数据结构

```
FileChange {
    file_path: string       // 文件路径（相对于项目根目录）
    change_type: added | modified | deleted | renamed
    old_path?: string       // renamed 时的原路径
    additions: number       // 新增行数
    deletions: number       // 删除行数
    diff_content: string    // Unified Diff 格式的变更内容
    language: string        // 文件语言（用于语法高亮）
}

ExecutionResult {
    execution_id: number
    files_changed: FileChange[]
    summary: string          // Agent 自身生成的变更摘要
    total_additions: number
    total_deletions: number
    files_count: number
}
```

## 3. Diff 预览

### 3.1 Diff 渲染方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| diff2html | 轻量、易集成、支持行内/并排视图 | 功能相对简单 |
| Monaco Editor Diff | 功能强大、VS Code 同款体验 | 体积较大 |
| 自定义渲染 | 完全可控 | 开发成本高 |

**建议**：MVP 用 diff2html，后续可考虑升级为 Monaco Diff Editor。

### 3.2 Diff 视图功能

| 功能 | 说明 |
|------|------|
| 并排视图 | 左旧右新的双栏对比 |
| 行内视图 | 单栏显示，删除行（红）和新增行（绿）交替 |
| 语法高亮 | 按文件语言高亮代码 |
| 文件导航 | 左侧文件树，点击跳转到对应文件的 Diff |
| 变更统计 | 每个文件的 +N / -N 行数统计 |
| 折叠未变更区域 | 只展示变更周围的上下文行 |
| 行级注释 | 可在特定行添加审核注释（进阶） |

## 4. 审批流程

### 4.1 审批操作

| 操作 | 说明 | 后续行为 |
|------|------|---------|
| **通过** | 接受所有变更 | 自动 Commit（或 Merge 分支） |
| **拒绝** | 不接受变更 | 回滚分支，任务返回待调度 |
| **部分采纳** | 接受部分文件的变更 | 选择性 Commit，其余回滚 |
| **返回修改** | 附加修改意见重新执行 | 将意见追加到 Prompt，重新调度 |

### 4.2 通过流程

```
用户点击 "通过并提交"
  ↓
生成 Commit Message（可编辑）
  ↓
执行 Git Commit
  ↓
可选：合并到主分支
  ↓
可选：推送到远程
  ↓
标记任务为 "已完成"
```

### 4.3 拒绝流程

```
用户点击 "拒绝并重做"
  ↓
弹出反馈输入框
  ├── 预设选项：代码质量差 / 不符合需求 / 有Bug / 风格不统一
  └── 自由文本：详细修改意见
  ↓
回滚 Git 分支到执行前状态
  ↓
任务状态 → "待调度"
  ↓
原 Prompt + 反馈意见 → 新的 Prompt
  ↓
等待用户决定：自动重新执行 / 手动修改后执行
```

### 4.4 部分采纳

- 以文件为粒度选择采纳/放弃
- 勾选需要保留的文件变更
- 未勾选的文件执行 `git checkout -- <file>` 恢复
- 生成的 Commit 只包含选中的文件

## 5. Git 集成

### 5.1 分支管理策略

```
main (或用户的当前分支)
  │
  ├── agent/task-42-login-feature    ← Agent 工作分支
  │     ├── commit: "agent: implement login feature"
  │     └── (审核通过后合并回 main)
  │
  ├── agent/task-43-fix-router-bug
  │     └── (审核中...)
  │
  └── agent/task-44-add-tests
        └── (执行中...)
```

### 5.2 分支命名规则

```
agent/{task_id}-{slugified_title}

示例：
- agent/42-implement-login-feature
- agent/43-fix-router-navigation-bug
- agent/44-add-unit-tests-for-auth
```

### 5.3 Git 操作清单

| 操作 | 时机 | 命令 |
|------|------|------|
| 创建分支 | 任务开始执行前 | `git checkout -b agent/task-{id}` |
| 收集变更 | 执行完成后 | `git diff` |
| 提交变更 | 审核通过 | `git add . && git commit` |
| 合并分支 | 审核通过后 | `git checkout main && git merge agent/task-{id}` |
| 删除分支 | 合并完成后 | `git branch -d agent/task-{id}` |
| 回滚变更 | 审核拒绝 | `git checkout main && git branch -D agent/task-{id}` |

### 5.4 冲突处理

```
合并时检测到冲突
  ↓
标记为 "合并冲突"
  ↓
展示冲突文件列表
  ↓
选项：
  ├── 手动解决：用户在外部编辑器解决后确认
  ├── 保留 Agent 版本：`git checkout --theirs`
  ├── 保留原有版本：`git checkout --ours`
  └── 放弃合并：`git merge --abort`
```

## 6. Commit Message 生成

### 6.1 自动生成规则

```
格式：
[agent] {type}: {title}

Agent: {agent_name}
Task: #{task_id}
Model: {model_name}

{agent_generated_summary}

示例：
[agent] feat: implement user login feature

Agent: Claude Code
Task: #42
Model: claude-sonnet-4-20250514

Added login/logout functionality with JWT token management.
- Created src/auth/login.ts with login() and logout() functions
- Updated router to add /login route
- Added LoginForm component with validation
```

### 6.2 用户可编辑

- 自动生成后弹出编辑框
- 用户可修改 Commit Message
- 支持快捷确认（Enter 直接提交）

## 7. 结果评分

### 7.1 评分维度

| 维度 | 说明 | 权重 |
|------|------|------|
| 正确性 | 代码是否正确实现了需求 | 40% |
| 代码质量 | 代码风格、可读性、最佳实践 | 25% |
| 完整性 | 是否覆盖了所有需求点 | 20% |
| 效率 | Token 消耗和执行时间是否合理 | 15% |

### 7.2 快速评分

- 审核通过时弹出 1-5 星快速评分
- 可选填评价文字
- 评分数据用于 Agent 能力画像的动态更新

### 7.3 评分数据应用

- 累计评分影响 Agent 的能力画像
- 低分 Agent + 任务类型组合会被标记提醒
- 高分组合会被推荐使用

## 8. 反馈循环

### 8.1 反馈增强 Prompt

```
原始 Prompt:
"实现用户登录功能，使用 JWT Token"

第一次执行 → 审核拒绝
反馈："缺少输入验证，密码需要 bcrypt 加密"

增强后的 Prompt:
"实现用户登录功能，使用 JWT Token

【历史反馈 - 请务必注意】
- 需要对用户输入进行验证（邮箱格式、密码长度）
- 密码必须使用 bcrypt 加密存储
- 上次执行的问题：缺少输入验证和密码加密"

第二次执行 → 审核通过 ✅
```

### 8.2 反馈模板

| 反馈类型 | 预设文案 |
|----------|---------|
| 不符合需求 | "请重新阅读需求，重点关注：{specific_point}" |
| 代码质量差 | "请改进代码质量：{quality_issue}" |
| 有 Bug | "以下场景存在 Bug：{bug_description}" |
| 缺少测试 | "请为以下功能添加测试：{test_scope}" |
| 风格不统一 | "请遵循项目的编码规范：{style_guide}" |

## 9. 数据库扩展

### 9.1 新增表：execution_reviews

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PRIMARY KEY | 自增主键 |
| execution_id | INTEGER NOT NULL | 关联执行 ID |
| action | TEXT NOT NULL | approve / reject / partial / revise |
| feedback | TEXT | 反馈内容 |
| selected_files | TEXT | JSON 格式的采纳文件列表（部分采纳时） |
| rating | INTEGER | 评分（1-5） |
| rating_comment | TEXT | 评分说明 |
| commit_hash | TEXT | 提交的 Git Hash（通过时） |
| reviewed_at | TEXT | 审核时间 |
