# 05 - 智能化能力

## 1. 概述

智能化能力是让调度系统从"工具"进化为"助手"的关键。它包括自动理解项目上下文、智能拆分复杂任务、根据历史数据选择最佳 Agent、以及用自然语言描述调度策略等能力。

## 2. 任务智能拆分

### 2.1 核心思路

用户输入一个高层需求（如"给项目添加用户认证系统"），系统调用 AI 将其拆分为多个可独立执行的 Agent 子任务。

### 2.2 拆分流程

```
用户输入高层需求
  ↓
系统注入项目上下文（技术栈、目录结构、已有代码概况）
  ↓
调用 AI 进行任务拆分
  ↓
生成子任务列表（含依赖关系）
  ↓
用户审核/调整子任务
  ↓
确认后批量创建 Agent 任务
```

### 2.3 拆分 Prompt 设计

```
你是一个任务拆分专家。请将以下需求拆分为多个独立的、可执行的开发任务。

【项目上下文】
- 技术栈：{tech_stack}
- 项目结构：{directory_tree}
- 现有功能概述：{existing_features}
- 编码规范：{coding_standards}

【需求描述】
{user_requirement}

【输出格式】
请以 JSON 格式输出，包含：
1. tasks: 任务数组，每个任务包含：
   - title: 任务标题
   - description: 详细描述
   - prompt: 建议的 Agent 指令
   - estimated_complexity: low / medium / high
   - recommended_agent: claude_code / codex
   - dependencies: 依赖的任务索引数组
2. execution_order: 建议的执行顺序说明
```

### 2.4 拆分示例

**输入**："给 Mini-Todo 添加用户认证系统"

**拆分结果**：

| # | 任务 | 复杂度 | 依赖 | 推荐 Agent |
|---|------|--------|------|-----------|
| 1 | 设计认证数据库表结构 | 低 | 无 | Claude Code |
| 2 | 实现 Rust 后端认证 API | 高 | #1 | Claude Code |
| 3 | 创建登录/注册 Vue 组件 | 中 | #1 | Claude Code |
| 4 | 实现前端路由守卫 | 低 | #3 | Codex |
| 5 | 添加 Token 管理和自动刷新 | 中 | #2, #3 | Claude Code |
| 6 | 编写认证模块单元测试 | 中 | #2, #3 | Claude Code |

### 2.5 用户交互

- 展示拆分结果后，用户可以：
  - 调整任务顺序和依赖关系
  - 修改单个任务的描述和 Prompt
  - 删除不需要的任务
  - 手动添加遗漏的任务
  - 更换推荐的 Agent
- 确认后一键创建所有子任务

## 3. 项目上下文注入

### 3.1 上下文类型

| 类型 | 来源 | 说明 |
|------|------|------|
| 技术栈信息 | package.json, Cargo.toml 等 | 自动检测项目使用的技术和依赖 |
| 目录结构 | 文件系统扫描 | 生成树状目录（排除 node_modules 等） |
| 项目文档 | CLAUDE.md, README.md 等 | 开发规范、架构说明 |
| 代码概况 | 静态分析 | 主要模块、入口文件、路由定义等 |
| Git 信息 | Git 仓库 | 最近的 Commit 记录、当前分支 |
| 已有任务 | 历史执行记录 | 之前执行过的相关任务和结果 |

### 3.2 上下文构建

```
ProjectContext {
    name: string                // 项目名称
    path: string                // 项目路径
    tech_stack: TechStack       // 技术栈信息
    directory_tree: string      // 目录树（过滤后）
    documentation: string[]     // 项目文档内容
    recent_commits: Commit[]    // 最近 N 条 Commit
    active_branch: string       // 当前分支
    custom_instructions: string // 用户自定义指令（如 CLAUDE.md）
}
```

### 3.3 上下文注入策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| 全量注入 | 将所有上下文拼接到 Prompt 开头 | 简单项目、首次任务 |
| 按需注入 | 根据任务类型选择相关上下文 | 大型项目、节省 Token |
| 分层注入 | 先注入概要，Agent 按需请求详情 | 超大项目 |

### 3.4 上下文缓存

- 项目上下文生成一次后缓存
- 检测到文件变更时增量更新
- 用户可手动刷新上下文
- 缓存有效期可配置（默认 1 小时）

## 4. 智能 Agent 分配

### 4.1 分配算法

```
输入：待执行任务
输出：推荐使用的 Agent

算法：
1. 获取任务的类型标签（如 refactor, test, bugfix）
2. 获取所有可用 Agent 的能力画像
3. 计算每个 Agent 的匹配分数：
   匹配分 = Σ (任务需求维度_i × Agent能力维度_i × 历史表现权重_i)
4. 考虑约束条件：
   - Agent 当前负载（是否已满载）
   - Agent 是否支持目标项目的技术栈
   - 成本预算约束
5. 返回匹配分最高的 Agent
```

### 4.2 匹配维度

| 任务特征 | 匹配的 Agent 能力 |
|----------|-----------------|
| 涉及多文件修改 | 多文件操作能力 |
| 需要理解复杂逻辑 | 代码理解深度 |
| 简单/明确的修改 | 执行速度 |
| 需要写测试 | 测试编写能力 |
| 重构任务 | 重构能力 |
| 有成本限制 | 成本效率 |

### 4.3 历史表现学习

```
AgentPerformance {
    agent_id: number
    task_type: string      // 任务类型
    success_rate: float    // 成功率（审核通过率）
    avg_rating: float      // 平均评分
    avg_cost: float        // 平均成本
    avg_duration: float    // 平均耗时
    sample_count: number   // 样本数量
    last_updated: datetime
}
```

- 每次任务完成（审核）后更新统计
- 样本数 < 5 时使用默认能力画像
- 样本数 >= 5 时逐步过渡到历史数据驱动

### 4.4 分配结果展示

```
推荐 Agent: Claude Code (匹配度 92%)
  理由：
  - 任务涉及 5+ 文件修改 → Claude Code 多文件能力: 优秀
  - 历史同类任务成功率: 87%
  - 预估成本: $0.05

  备选：
  - Codex (匹配度 65%) - 速度更快但多文件能力较弱
```

## 5. 自然语言调度

### 5.1 概念

允许用户用自然语言描述调度策略，系统自动转化为调度规则。

### 5.2 示例

| 用户输入 | 系统理解 |
|---------|---------|
| "每天早上 9 点检查项目中的 TODO 注释" | Cron: `0 9 * * *`，Prompt: "扫描所有文件中的 TODO/FIXME 注释并生成报告" |
| "当 task-42 完成后立即开始测试任务" | 依赖: finish-to-start(#42)，自动触发 |
| "用 Claude Code 重构 auth 模块，完成后用 Codex 写测试" | 创建两个任务，设置依赖，分别分配 Agent |
| "低优先级，明天处理就行" | 优先级: low，scheduled_at: tomorrow 9:00 |

### 5.3 实现方式

- 使用 AI 解析自然语言，提取调度意图
- 转化为结构化的调度参数
- 展示解析结果让用户确认
- 确认后创建任务和调度规则

### 5.4 快捷指令

预设一些常用的自然语言模式：

| 模式 | 触发词 |
|------|--------|
| 快速任务 | "快速 {prompt}" → 立即创建并执行 |
| 定时任务 | "每天/每周 {时间} {prompt}" → 创建定时任务 |
| 串行任务 | "先...然后...最后..." → 创建依赖链 |
| 批量任务 | "对所有 {条件} 的项目执行 {prompt}" → 批量创建 |

## 6. 智能建议

### 6.1 任务创建建议

- 根据项目的 Git 历史分析常见修改模式
- 基于 TODO/FIXME 注释自动建议任务
- 检测依赖过时自动建议升级任务
- 代码质量分析后建议重构任务

### 6.2 Prompt 优化建议

- 分析用户编写的 Prompt
- 建议补充缺失的上下文
- 提示可能的歧义或遗漏
- 推荐使用更精确的描述

### 6.3 执行策略建议

- 根据任务复杂度推荐合适的 Agent 和模型
- 基于历史数据预估执行时间和成本
- 建议是否需要拆分为子任务
- 提醒可能的风险（如大范围修改建议先备份）

## 7. 多项目管理

### 7.1 项目注册

```
Project {
    id: number
    name: string              // 项目名称
    path: string              // 本地路径
    tech_stack: string[]      // 技术栈标签
    description: string       // 项目描述
    instructions_file: string // 指令文件路径（如 CLAUDE.md）
    default_agent_id: number  // 默认 Agent
    max_concurrent: number    // 最大并行任务数
    created_at: datetime
}
```

### 7.2 项目仪表盘

- 项目列表总览
- 每个项目显示：活跃任务数、待审核数、累计成本
- 快速切换项目上下文
- 项目级别的执行历史和统计

## 8. 统计仪表盘

### 8.1 核心指标

| 指标 | 说明 |
|------|------|
| 任务完成率 | 审核通过数 / 总执行数 |
| 平均执行时间 | 所有任务的平均执行耗时 |
| 总成本 | 累计 Token 费用 |
| 一次通过率 | 首次执行就审核通过的比率 |
| Agent 利用率 | Agent 活跃执行时间 / 总可用时间 |

### 8.2 图表类型

| 图表 | 展示内容 |
|------|---------|
| 折线图 | 日/周/月的任务执行数量趋势 |
| 饼图 | 任务状态分布（成功/失败/取消） |
| 柱状图 | 各 Agent 的任务数量和成功率对比 |
| 热力图 | 任务执行的时间分布（哪些时段最活跃） |
| 成本曲线 | 日/周/月的成本趋势 |
