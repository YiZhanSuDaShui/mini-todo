# 智能化能力 - 总体规划与设计决策

## 1. 开发目标

基于 [05-智能化能力.md](../05-智能化能力.md) 灵感文档，结合讨论结果，本次只实现**大任务智能拆分**功能：

1. 用户输入一个高层需求，系统调用 AI 将其拆分为多个可独立执行的 Agent 子任务
2. 自动构建项目上下文并注入到拆分 Prompt 中
3. 拆分结果可供用户审核、调整后一键创建子任务
4. 拆分出的子任务自动关联依赖关系和推荐 Agent

## 2. 排除功能

| 功能 | 决策 | 原因 |
|------|------|------|
| Token 消耗统计 | 不实现 | 当前阶段不需要 |
| 自然语言调度 | 不实现 | 复杂度高，收益不明确 |
| 智能建议 | 不实现 | 当前阶段不需要 |
| 多项目管理 | 不实现 | 增加 UI 复杂度 |
| 统计仪表盘 | 不实现 | 当前阶段不需要 |
| 智能 Agent 分配 | 不实现 | 只有 Claude Code 和 Codex 两个 Agent，无需算法分配 |

## 3. 与现有架构的关系

### 3.1 现有基础设施

```
当前体系：
  Todo （大任务）
    ├── agent_id         → 绑定的 Agent
    ├── agent_project_path → 项目目录
    └── Subtask （子任务）
          ├── title, content  → Agent 执行指令的来源
          ├── schedule_status → 调度状态（none/pending/queued/running/...）
          └── agent_executions → 执行记录

  PromptTemplate → 模板系统（已有 5 个内置模板）
  TaskDependency → 子任务依赖关系（已实现）
  TaskScheduler  → 调度引擎（已实现）
```

### 3.2 新增扩展

```
新增：
  任务拆分服务（task_splitter）
    ├── 项目上下文构建器（context_builder）
    │     ├── 读取 package.json / Cargo.toml 等检测技术栈
    │     ├── 生成目录树（过滤 node_modules 等）
    │     ├── 读取 CLAUDE.md / README.md 等项目文档
    │     └── 获取 Git 信息（分支、最近提交）
    ├── 拆分引擎（splitter_engine）
    │     ├── 构建拆分 Prompt（上下文 + 需求 + 输出格式）
    │     ├── 调用 Agent CLI 执行拆分
    │     └── 解析 JSON 输出为拆分结果
    └── 结果应用器
          ├── 批量创建子任务
          ├── 创建依赖关系
          └── 设置推荐 Agent 和优先级

  前端组件
    ├── TaskSplitDialog → 任务拆分对话框
    └── SplitResultPanel → 拆分结果预览和调整面板
```

## 4. 实现阶段划分

### Phase 1：项目上下文构建器

**目标**：能根据项目路径自动生成结构化的项目上下文信息。

| 任务 | 说明 |
|------|------|
| 技术栈检测 | 读取 package.json、Cargo.toml、go.mod 等文件，提取依赖信息 |
| 目录树生成 | 递归扫描目录，排除 node_modules/target/.git 等，限制深度 |
| 文档提取 | 读取 CLAUDE.md、README.md、AGENTS.md 等项目说明文件 |
| Git 信息 | 当前分支、最近 N 条 commit 的概要 |
| 上下文序列化 | 将所有信息组装为可注入 Prompt 的文本块 |

### Phase 2：拆分引擎

**目标**：将高层需求 + 项目上下文传入 Agent CLI，获取结构化的拆分结果。

| 任务 | 说明 |
|------|------|
| 拆分 Prompt 模板 | 设计系统 Prompt，定义输出 JSON 格式 |
| Agent CLI 调用 | 复用现有 `create_command` 和 `run_process` 流程，调用 Claude Code CLI |
| 结果解析 | 从 Agent 输出中提取 JSON 格式的任务列表 |
| 错误处理 | 处理 AI 输出格式不合规、超时、解析失败等情况 |

### Phase 3：前端交互

**目标**：提供完整的拆分 → 预览 → 调整 → 创建交互流程。

| 任务 | 说明 |
|------|------|
| 拆分触发入口 | 在 EditorView 中添加"智能拆分"按钮 |
| 需求输入 | 对话框中输入高层需求描述 |
| 拆分进度 | 显示 AI 正在拆分的加载状态 |
| 结果预览 | 列表展示拆分出的子任务，含标题、描述、复杂度、依赖关系 |
| 结果调整 | 支持编辑/删除/重排子任务，修改依赖关系 |
| 一键创建 | 确认后批量创建子任务和依赖关系 |

## 5. 技术选型

| 层级 | 选择 | 说明 |
|------|------|------|
| AI 调用 | Claude Code CLI | 复用现有的 `create_command` + `-p` 模式调用 |
| 输出格式 | `--output-format json` | 不需要流式，只需最终结果 |
| 结果解析 | `serde_json` | 从 AI 文本输出中提取 JSON |
| 前端组件 | Element Plus Dialog + Table | 保持与现有 UI 风格一致 |

## 6. 关键设计要点

### 6.1 拆分粒度

每个拆分出的子任务应满足：
- **独立可执行**：单个 Agent 可独立完成，无需人工干预
- **清晰边界**：修改范围明确，不与其他子任务冲突
- **可验证**：有明确的完成标准

### 6.2 上下文大小控制

- 目录树限制深度（默认 3 层）
- 文档内容截取前 N 行（防止超大 README）
- Git commit 仅取最近 10 条
- 总上下文控制在 4000 字以内

### 6.3 拆分结果的 JSON Schema

```json
{
  "tasks": [
    {
      "title": "任务标题",
      "description": "详细描述",
      "prompt": "建议的 Agent 执行指令",
      "complexity": "low | medium | high",
      "recommended_agent": "claude_code | codex",
      "dependencies": [0, 1]
    }
  ],
  "summary": "整体执行策略说明"
}
```

### 6.4 与现有调度系统的集成

拆分创建的子任务直接进入现有的调度体系：
- `schedule_status` 默认为 `none`（手动模式）或 `pending`（自动模式）
- 依赖关系通过 `task_dependencies` 表管理
- 可利用现有的优先级队列和并发控制

## 7. 风险分析

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| AI 拆分质量不稳定 | 子任务粒度不合理 | 用户审核环节 + 重新拆分按钮 |
| AI 输出格式不可控 | JSON 解析失败 | 多次重试 + 宽松解析（从文本中提取 JSON 块） |
| 上下文过大导致慢/贵 | 拆分耗时长、Token 消耗高 | 上下文大小限制 + 按需注入 |
| 依赖关系不准确 | 执行顺序错误 | 用户可手动调整依赖 |
