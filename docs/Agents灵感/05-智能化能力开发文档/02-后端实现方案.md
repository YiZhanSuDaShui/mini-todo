## 2. 项目上下文构建器（context_builder.rs）

### 2.1 数据结构

```rust
pub struct ProjectContext {
    pub name: String,

    pub category: String,   // "frontend" | "backend" | "database" | "tool"
    pub name: String,       // "Vue 3", "Rust", "SQLite"

    pub filename: String,
    pub content: String,    // 截取前 200 行
}

pub struct CommitInfo {
    pub hash: String,       // 短 hash
    pub message: String,
    pub date: String,
}
```

### 2.2 技术栈检测

通过检测项目目录下的特征文件来识别技术栈：

| 文件 | 技术 | 提取信息 |
|------|------|---------|
| `package.json` | Node.js 生态 | dependencies 中的框架（vue, react, next, express 等） |
| `Cargo.toml` | Rust | `[dependencies]` 中的核心 crate |
| `go.mod` | Go | module 名称 |
| `requirements.txt` / `pyproject.toml` | Python | 主要依赖 |
| `*.csproj` | .NET | TargetFramework 和 PackageReference |
| `pubspec.yaml` | Flutter/Dart | dependencies |
| `tsconfig.json` | TypeScript | 标记项目使用 TypeScript |

实现思路：
- 使用 `std::fs::read_to_string` 读取文件
- 使用简单的字符串解析（不需要完整的 TOML/JSON parser 来提取关键信息，`serde_json` 已可用）
- `package.json` 用 `serde_json` 解析，提取 `dependencies` 和 `devDependencies` 中的知名框架
- `Cargo.toml` 用 `toml` crate 解析（或简单字符串匹配 `[dependencies]` 段落）

### 2.3 目录树生成

```rust
pub fn generate_directory_tree(
    project_path: &Path,
    max_depth: usize,       // 默认 3
    max_entries: usize,      // 默认 200
) -> String
```

实现思路：
- 递归遍历目录，排除：`.git`, `node_modules`, `target`, `dist`, `build`, `__pycache__`, `.next`, `.nuxt`, `vendor`
- 仅展示文件名和目录结构，不读取文件内容
- 使用 `├──` `└──` `│` 字符绘制树状结构
- 超过 `max_entries` 时截断并标注 `... (N more items)`
- 使用 `tokio::task::spawn_blocking` 执行（复用 triggers.rs 的经验）

### 2.4 文档提取

```rust
const DOC_FILES: &[&str] = &[
```

### 2.5 Git 信息获取

```rust
pub fn get_git_info(project_path: &Path) -> Option<GitInfo>
```

实现思路：
- 复用 `triggers.rs` 中的 `get_git_head` 模式
- `git rev-parse --abbrev-ref HEAD` → 当前分支
- `git log --oneline -10` → 最近 10 条 commit
- `std::process::Command` 同步调用（操作极快）

### 2.7 上下文大小控制

按照优先级控制总文本大小，确保不超过 4000 字符：

1. 技术栈信息（必含，通常 < 200 字）
2. 目录树（可缩减深度）
3. 项目文档（可缩减行数）
4. Git 信息（可减少 commit 数）

如果总长度超限，依次缩减 4 → 3 → 2。

## 3. 任务拆分引擎（task_splitter.rs）

### 3.1 数据结构

```rust
pub struct SplitRequest {
    pub todo_id: i64,              // 关联的 Todo ID
    pub requirement: String,        // 用户输入的高层需求
    pub project_path: String,       // 项目目录
    pub agent_type: String,         // 使用哪个 Agent 做拆分（默认 claude_code）
}

pub struct SplitResult {
    pub tasks: Vec<SplitTask>,
    pub summary: String,           // 整体执行策略说明
}

pub struct SplitTask {
    pub title: String,
    pub description: String,
    pub prompt: String,            // 建议的 Agent 执行指令
    pub complexity: String,        // "low" | "medium" | "high"
    pub recommended_agent: String, // "claude_code" | "codex"
    pub dependencies: Vec<usize>,  // 依赖的任务索引（0-based）
}
```

### 3.2 拆分 Prompt 设计

```
你是一个任务拆分专家。请将以下需求拆分为多个独立的、可由 AI 编程助手执行的开发子任务。

【项目上下文】
{context_text}

【需求描述】
{requirement}

【拆分原则】
1. 每个子任务必须是单个 Agent 可独立完成的，修改范围明确
2. 子任务之间不应有代码冲突
3. 正确标注依赖关系，被依赖的任务排在前面
4. prompt 字段应包含足够的细节让 Agent 直接执行

【输出要求】
请严格以 JSON 格式输出，不要包含任何额外文字：
{
  "tasks": [
    {
      "title": "简洁的任务标题",
      "description": "任务的详细描述，包含修改哪些文件、实现什么功能",
      "prompt": "给 Agent 的具体执行指令",
      "complexity": "low | medium | high",
      "recommended_agent": "claude_code | codex",
      "dependencies": []
    }
  ],
  "summary": "整体执行策略的简要说明"
}
```

### 3.3 Agent 调用流程

```
构建上下文
  ↓
组装 Prompt（系统 Prompt + 上下文 + 需求）
  ↓
调用 Agent CLI（claude -p "..." --output-format json）
  ↓
从输出中提取 JSON
  ↓
验证 JSON 结构
  ↓
返回 SplitResult
```

实现思路：
- 复用 `create_command` 构建 Claude Code 命令
- 使用 `--output-format json` 而非 `stream-json`（只需最终结果）
- 不使用 `--dangerously-skip-permissions`（拆分任务不需要执行代码）
- 从 stdout 读取完整输出
- 用正则 `\{[\s\S]*\}` 从输出文本中提取 JSON 块（AI 可能在 JSON 前后添加说明文字）
- 用 `serde_json` 解析为 `SplitResult`
- 超时设置 120 秒（拆分通常较快）

### 3.4 JSON 提取策略

AI 输出可能不是纯 JSON，需要从文本中提取：

```rust
    // 策略 3：查找最外层的 { ... } 对
    // 从第一个 { 到最后一个 } 的内容

```

### 3.5 错误处理

| 错误场景 | 处理方式 |
|---------|---------|
| Agent CLI 不可用 | 返回明确错误提示 |
| 调用超时 | 120 秒超时，提示重试 |
| JSON 解析失败 | 尝试多种提取策略，仍失败则返回原始文本供用户参考 |
| 任务数量为 0 | 提示需求可能过于简单，不需要拆分 |
| 依赖索引越界 | 自动修正为空依赖 |

## 4. 结果应用（批量创建子任务）

### 4.1 应用流程

```
用户确认拆分结果
  ↓
按依赖顺序排列任务
  ↓
逐个创建子任务（create_subtask）
  ↓
建立索引映射（拆分结果索引 → 子任务 ID）
  ↓
创建依赖关系（add_dependency）
  ↓
根据 Todo 的 schedule_strategy 设置 schedule_status
```

### 4.2 Tauri 命令设计

```rust
/// 构建项目上下文（可单独调用，用于预览）

pub async fn apply_split_result(

```
