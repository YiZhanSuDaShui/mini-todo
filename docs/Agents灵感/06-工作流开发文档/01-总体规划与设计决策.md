# 工作流引擎 - 总体规划与设计决策

## 1. 开发目标

为每个 Todo（大任务）提供**顺序编排的工作流**能力，用户可以定义一条执行流水线，指定各步骤按顺序依次执行，直到全部完成。

核心需求：
1. 给大任务配置一条工作流（有序步骤列表）
2. 每个步骤可以是「执行某个子任务」或「执行自定义提示词」
3. 工作流按步骤顺序自动推进：步骤1完成 → 步骤2 → ... → 结束
4. 不需要可视化流程编排，在现有界面中配置即可

## 2. 排除功能

| 功能 | 决策 | 原因 |
|------|------|------|
| 可视化工作流编辑器（拖拽连线） | 不实现 | 简单列表配置即可 |
| 条件判断 / 分支网关 | 不实现 | 仅需纯顺序执行 |
| 并行网关 | 不实现 | 当前无需并行分支 |
| 延时节点 / 定时触发 | 不实现 | 现有调度系统已覆盖 |
| 工作流模板 / 导入导出 | 不实现 | 首版不需要 |
| 数据流变量传递 | 不实现 | 步骤间无需复杂数据传递 |
| Shell 命令步骤 | 不实现 | 首版聚焦子任务和提示词 |

## 3. 核心概念

### 3.1 工作流（Workflow）

工作流是附加在 Todo 上的**有序步骤列表**，定义了该大任务下各操作的执行顺序。

```
Todo: "重构用户模块"
  ├── Subtask A: "设计新数据库 schema"
  ├── Subtask B: "编写后端 API"
  └── Subtask C: "编写前端页面"

工作流:
  Step 0: [subtask] 执行子任务 A（设计新数据库 schema）
  Step 1: [prompt]  "请 review 上述数据库 schema，列出潜在问题"
  Step 2: [subtask] 执行子任务 B（编写后端 API）
  Step 3: [subtask] 执行子任务 C（编写前端页面）
  Step 4: [prompt]  "请对整个重构做最终检查，确认无遗漏"
```

### 3.2 步骤类型

| 类型 | 标识 | 说明 |
|------|------|------|
| 执行子任务 | `subtask` | 引用一个已有的子任务，调度 Agent 执行 |
| 执行提示词 | `prompt` | 用户输入的自定义提示词，调度 Agent 执行 |

### 3.3 步骤状态

| 状态 | 说明 |
|------|------|
| `pending` | 未执行，等待前序步骤完成 |
| `running` | 正在执行 |
| `completed` | 执行完成 |
| `failed` | 执行失败 |
| `skipped` | 被跳过（手动跳过或关联子任务已删除） |

### 3.4 工作流状态

工作流整体通过 `workflow_current_step` 字段追踪当前执行到哪一步：
- `-1`：工作流未启动
- `0 ~ N-1`：当前正在执行第 N 步
- `>= 总步骤数`：工作流已全部完成

## 4. 与现有架构的关系

### 4.1 现有基础设施

```
当前体系：
  Todo
    ├── agent_id, agent_project_path → Agent 配置
    ├── schedule_strategy → 调度策略（manual/auto/cron/...）
    └── Subtask
          ├── schedule_status → none/pending/queued/running/completed/failed/cancelled
          ├── 依赖关系 → task_dependencies 表
          └── Agent 执行 → agent_executions 表

  TaskScheduler（engine.rs）
    ├── execute_task() → 执行单个子任务
    │     └── 完成后调用 trigger_downstream()
    └── handle_task_failure() → 失败重试逻辑
```

### 4.2 工作流扩展

```
新增：
  workflow_steps 表
    ├── 每行 = 一个步骤
    ├── step_order = 执行顺序
    ├── step_type = subtask 或 prompt
    └── status = 步骤自身状态

  Todo 表扩展
    ├── workflow_enabled → 是否启用工作流
    └── workflow_current_step → 当前执行到第几步

  services/scheduler/workflow.rs（新增模块）
    ├── start_workflow() → 启动工作流，执行第一步
    ├── advance_workflow() → 推进到下一步
    ├── execute_prompt_step() → 执行提示词步骤
    └── on_step_completed() → 步骤完成回调
```

### 4.3 与现有调度的关系

| 场景 | 行为 |
|------|------|
| Todo **未启用**工作流 | 子任务独立调度，使用现有逻辑（依赖、手动/自动调度等） |
| Todo **启用**工作流 | 子任务的执行顺序由工作流定义，不使用依赖关系调度 |

启用工作流后，子任务的调度由工作流引擎接管：
- 子任务不再独立进入 pending/queued
- 工作流引擎根据步骤顺序，逐个将子任务提交到调度器
- 当前步骤完成后，自动推进下一步

### 4.4 替换 post_action

现有的 `post_action` 字段（`none/git_commit/review/git_commit_and_review`）将被工作流机制取代：
- 原来用 post_action 实现的「Git Commit 后续动作」和「人工审核」可以在未来作为新的步骤类型加入工作流
- 当前版本先移除 post_action，聚焦于 subtask 和 prompt 两种步骤类型
- `reviewing` 状态暂时保留在状态机中，但不在首版工作流中使用

## 5. 实现阶段

### Phase 1：数据层

| 任务 | 说明 |
|------|------|
| 数据库迁移 | 创建 `workflow_steps` 表 |
| 数据库迁移 | `todos` 表新增 `workflow_enabled`、`workflow_current_step` |
| 数据库迁移 | 移除 `post_action` 字段（或标记废弃） |
| 模型更新 | Rust 数据模型和前端类型同步 |
| CRUD 命令 | 工作流步骤的增删改查 Tauri 命令 |

### Phase 2：执行引擎

| 任务 | 说明 |
|------|------|
| workflow.rs 模块 | 工作流引擎核心逻辑 |
| 启动工作流 | start_workflow：设置 current_step=0，执行第一步 |
| 步骤推进 | advance_workflow：当前步骤完成后推进到下一步 |
| 子任务步骤执行 | 将关联的子任务提交到调度器 |
| 提示词步骤执行 | 调用 Agent 执行自定义提示词 |
| 引擎集成 | engine.rs 中子任务完成时检查工作流并推进 |

### Phase 3：前端

| 任务 | 说明 |
|------|------|
| 类型定义 | 前端 TypeScript 类型更新 |
| 工作流配置 UI | EditorView 中添加工作流步骤编辑 |
| 步骤管理 | 添加/删除/排序步骤 |
| 执行状态展示 | 显示当前工作流进度和各步骤状态 |
| 启动/停止控制 | 启动工作流、暂停、重置等操作 |

## 6. 关键设计要点

### 6.1 步骤与子任务的关系

- 一个 `subtask` 类型的步骤引用一个已有的子任务（通过 `subtask_id`）
- 一个子任务最多被一个步骤引用
- 未被工作流引用的子任务不受工作流影响，仍可独立调度
- 如果被引用的子任务被删除，对应步骤自动标记为 `skipped`

### 6.2 提示词步骤的执行

- 使用 Todo 级别配置的 Agent（`agent_id`）和项目路径（`agent_project_path`）
- 执行方式与子任务一致，通过 `AgentManager.start_background_execution()`
- 执行结果记录在 `agent_executions` 表中（`subtask_id` 为空或使用特殊标记）

### 6.3 工作流的启动与控制

| 操作 | 说明 |
|------|------|
| 启动 | 设置 `workflow_current_step = 0`，执行第一步 |
| 暂停 | 停止自动推进，当前执行中的步骤继续完成 |
| 继续 | 恢复自动推进 |
| 重置 | 将所有步骤状态重置为 pending，current_step 回到 -1 |
| 跳过当前步骤 | 将当前步骤标记为 skipped，推进到下一步 |

### 6.4 错误处理

| 场景 | 处理 |
|------|------|
| 步骤执行失败 | 步骤标记 failed，工作流暂停，等待用户决定 |
| 子任务引用不存在 | 步骤标记 skipped，自动推进下一步 |
| Agent 未配置 | 启动工作流时前置检查，给出提示 |
| 所有步骤完成 | 工作流标记完成 |

## 7. 风险分析

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 工作流与独立调度冲突 | 同一子任务同时被工作流和手动调度触发 | 启用工作流后禁用子任务的独立调度 |
| 长时间运行的步骤阻塞 | 后续步骤一直等待 | 支持跳过当前步骤 |
| 提示词步骤无关联子任务 | 执行记录难以归属 | 在 agent_executions 中标记来源为 workflow |
| 编辑工作流时有步骤在执行 | 顺序变更导致混乱 | 执行中的工作流不允许编辑步骤 |
