## 2. 数据库迁移

### 2.1 清理 post_action（迁移 v19）

由于 SQLite 不原生支持 DROP COLUMN（3.35.0+ 才支持），且 `post_action` 字段保留不影响功能，
采用**软废弃**策略：不删除列，但代码中不再读写。

### 2.2 创建 workflow_steps 表（迁移 v19）

```sql
CREATE TABLE IF NOT EXISTS workflow_steps (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    todo_id     INTEGER NOT NULL,
    step_order  INTEGER NOT NULL,
    step_type   TEXT NOT NULL CHECK(step_type IN ('subtask', 'prompt')),
    subtask_id  INTEGER,
    prompt_text TEXT,
    status      TEXT NOT NULL DEFAULT 'pending',
    created_at  TEXT NOT NULL DEFAULT (datetime('now', 'localtime')),
    FOREIGN KEY (todo_id) REFERENCES todos(id) ON DELETE CASCADE,
    FOREIGN KEY (subtask_id) REFERENCES subtasks(id) ON DELETE SET NULL
);
CREATE INDEX IF NOT EXISTS idx_workflow_steps_todo ON workflow_steps(todo_id, step_order);
```

### 2.3 Todo 表新增工作流字段（迁移 v19）

```sql
ALTER TABLE todos ADD COLUMN workflow_enabled INTEGER NOT NULL DEFAULT 0;
ALTER TABLE todos ADD COLUMN workflow_current_step INTEGER NOT NULL DEFAULT -1;
```

## 5. 工作流引擎（workflow.rs）

### 5.1 核心函数

```rust
/// 启动工作流：设置 current_step=0，执行第一步
pub async fn start_workflow(
    app: &tauri::AppHandle,
    todo_id: i64,
) -> Result<(), String>

/// 推进工作流到下一步
/// 由 engine.rs 在子任务完成时调用，或提示词步骤完成后调用
pub async fn advance_workflow(
    app: &tauri::AppHandle,
    todo_id: i64,
) -> Result<(), String>

/// 执行当前步骤
async fn execute_step(
    app: &tauri::AppHandle,
    todo_id: i64,
    step: &WorkflowStep,
) -> Result<(), String>

/// 暂停工作流（不中断当前执行中的步骤）
pub fn pause_workflow(app: &tauri::AppHandle, todo_id: i64) -> Result<(), String>

/// 重置工作流（所有步骤回到 pending，current_step = -1）
pub fn reset_workflow(app: &tauri::AppHandle, todo_id: i64) -> Result<(), String>

/// 跳过当前步骤
pub async fn skip_current_step(
    app: &tauri::AppHandle,
    todo_id: i64,
) -> Result<(), String>
```

### 5.2 启动工作流逻辑

```
start_workflow(todo_id):
  1. 检查 Todo 是否配置了 Agent（agent_id 非空）
  2. 获取工作流步骤列表
  3. 如果步骤为空 → 返回错误
  4. 设置 workflow_current_step = 0
  5. 设置 workflow_enabled = 1
  6. 将第一个步骤标记为 running
  7. 调用 execute_step(step[0])
```

### 5.3 步骤执行逻辑

```
execute_step(step):
  if step.step_type == "subtask":
    1. 检查 subtask_id 是否有效
    2. 如果子任务已删除 → 标记步骤 skipped，advance_workflow
    3. 将子任务状态设为 pending → 进入调度器队列
    4. 调度器自动拾取执行
    5. 执行完成后由 engine.rs 回调 advance_workflow

  if step.step_type == "prompt":
    1. 从 Todo 读取 agent_id 和 agent_project_path
    2. 构建 prompt（直接使用 step.prompt_text）
    3. 调用 AgentManager.start_background_execution()
    4. 使用与 execute_task 类似的轮询等待逻辑
    5. 完成后标记步骤 completed，调用 advance_workflow
```

### 5.4 推进逻辑

```
advance_workflow(todo_id):
  1. 读取当前 workflow_current_step
  2. 将当前步骤标记为 completed（如果还是 running）
  3. current_step += 1
  4. 如果 current_step >= 步骤总数：
     → 工作流完成，记录日志
     → 返回
  5. 获取下一步骤
  6. 如果下一步骤的 subtask_id 无效 → 标记 skipped，继续 advance
  7. 标记下一步骤为 running
  8. 调用 execute_step(下一步骤)
```

## 6. 引擎集成（engine.rs）

### 6.1 修改 execute_task 完成分支

在 `execute_task` 中，当子任务执行完成（status == "completed"）时：

```
现有逻辑：
  completed → mark_completed → trigger_downstream

新增逻辑：
  completed →
    1. 查询该子任务是否属于工作流步骤（workflow_db::find_step_by_subtask）
    2. 如果是：
       a. 标记子任务 completed
       b. 调用 workflow::advance_workflow(todo_id)
       c. 不调用 trigger_downstream（由工作流接管）
    3. 如果不是：
       a. 保持现有逻辑（mark_completed + trigger_downstream）
```

### 6.2 移除 post_actions 调用

移除 `execute_task` 中对 `post_actions::execute_post_action` 的调用，
改为工作流检查逻辑。
