# 06 - 工作流引擎

## 1. 概述

工作流引擎是系统的高级能力，允许用户将多个 Agent 任务编排为自动化流水线。通过可视化编辑器定义工作流，系统自动按照流程执行、在需要时暂停等待人工审核，最终完成复杂的多步骤任务。

## 2. 核心概念

### 2.1 术语定义

| 术语 | 说明 |
|------|------|
| **工作流 (Workflow)** | 一组按特定逻辑编排的节点和连线，定义了自动化执行的流程 |
| **节点 (Node)** | 工作流中的单个执行单元（如 Agent 任务、人工审核、条件判断） |
| **连线 (Edge)** | 节点之间的连接，定义执行顺序和数据流向 |
| **实例 (Instance)** | 工作流的一次具体执行，包含运行时状态和数据 |
| **触发器 (Trigger)** | 启动工作流的条件（手动、定时、事件） |

### 2.2 工作流定义结构

```json
{
  "id": "wf-001",
  "name": "Feature 开发流水线",
  "version": 1,
  "trigger": {
    "type": "manual"
  },
  "variables": {
    "feature_desc": { "type": "string", "required": true, "label": "功能描述" },
    "project_path": { "type": "path", "required": true, "label": "项目路径" }
  },
  "nodes": [...],
  "edges": [...]
}
```

## 3. 节点类型

### 3.1 Agent 任务节点

执行 AI Agent 的代码任务。

```json
{
  "id": "node-1",
  "type": "agent_task",
  "name": "编写功能代码",
  "config": {
    "agent_id": "auto",
    "prompt_template": "feature_development",
    "variables_mapping": {
      "feature_desc": "{{workflow.feature_desc}}",
      "project_path": "{{workflow.project_path}}"
    },
    "timeout_seconds": 300,
    "max_retries": 1
  }
}
```

### 3.2 人工审核节点

暂停工作流，等待人工审核确认。

```json
{
  "id": "node-2",
  "type": "human_review",
  "name": "代码审核",
  "config": {
    "review_type": "diff",
    "prompt": "请审核 Agent 生成的代码是否符合需求",
    "timeout_hours": 24,
    "actions": ["approve", "reject", "revise"]
  }
}
```

### 3.3 条件判断节点

根据条件决定工作流走向。

```json
{
  "id": "node-3",
  "type": "condition",
  "name": "检查测试结果",
  "config": {
    "expression": "{{prev_node.exit_code}} == 0",
    "true_branch": "node-4",
    "false_branch": "node-5"
  }
}
```

### 3.4 Shell 命令节点

执行本地 Shell 命令（如运行测试、构建项目）。

```json
{
  "id": "node-4",
  "type": "shell_command",
  "name": "运行测试",
  "config": {
    "command": "npm test",
    "working_directory": "{{workflow.project_path}}",
    "timeout_seconds": 120,
    "success_exit_codes": [0]
  }
}
```

### 3.5 通知节点

发送通知（系统通知、写入日志等）。

```json
{
  "id": "node-5",
  "type": "notification",
  "name": "通知完成",
  "config": {
    "type": "system",
    "title": "工作流完成",
    "message": "Feature '{{workflow.feature_desc}}' 开发流水线已完成"
  }
}
```

### 3.6 并行网关节点

将流程分为多个并行分支，或等待多个分支完成后汇合。

```json
{
  "id": "node-6",
  "type": "parallel_gateway",
  "name": "并行执行",
  "config": {
    "mode": "fork",
    "branches": ["node-7", "node-8", "node-9"]
  }
}
```

### 3.7 延时节点

暂停工作流指定时间后继续。

```json
{
  "id": "node-10",
  "type": "delay",
  "name": "等待 5 分钟",
  "config": {
    "duration_seconds": 300
  }
}
```

## 4. 连线与数据流

### 4.1 连线定义

```json
{
  "edges": [
    { "from": "node-1", "to": "node-2", "label": "执行完成" },
    { "from": "node-2", "to": "node-3", "label": "审核通过", "condition": "action == 'approve'" },
    { "from": "node-2", "to": "node-1", "label": "审核拒绝", "condition": "action == 'reject'" },
    { "from": "node-3", "to": "node-4", "label": "测试通过" },
    { "from": "node-3", "to": "node-5", "label": "测试失败" }
  ]
}
```

### 4.2 数据传递

- 上游节点的输出可作为下游节点的输入
- 使用 `{{node_id.output_field}}` 语法引用
- 工作流级变量使用 `{{workflow.variable_name}}` 引用
- 支持简单的表达式计算

### 4.3 上下文累积

```
工作流开始
  ↓ context = { workflow_vars }
node-1 执行 → context += { node_1_output, node_1_files_changed }
  ↓
node-2 审核 → context += { node_2_action, node_2_feedback }
  ↓
node-3 执行 → 可以引用 context 中的所有数据
```

## 5. 可视化编辑器

### 5.1 编辑器功能

| 功能 | 说明 |
|------|------|
| 拖拽节点 | 从节点面板拖拽节点到画布 |
| 连线 | 从节点输出端口拖拽到另一节点输入端口 |
| 配置面板 | 点击节点弹出右侧配置面板 |
| 缩放/平移 | 鼠标滚轮缩放，拖拽画布平移 |
| 自动布局 | 一键自动排列节点位置 |
| 撤销/重做 | Ctrl+Z / Ctrl+Y |
| 保存/加载 | 工作流定义的保存和加载 |

### 5.2 技术选型

| 方案 | 优点 | 缺点 |
|------|------|------|
| Vue Flow (vue-flow) | Vue 生态、功能丰富、社区活跃 | 学习曲线 |
| LogicFlow | 国产、流程图专用 | 生态较小 |
| JointJS | 功能强大、商业级 | 需商业授权 |
| 自绘 Canvas/SVG | 完全可控 | 开发成本极高 |

**建议**：使用 Vue Flow（@vue-flow/core），与现有 Vue 3 技术栈无缝集成。

## 6. 工作流执行引擎

### 6.1 执行流程

```
工作流实例创建
  ↓
填充工作流变量
  ↓
从起始节点开始执行
  ↓
┌─ 循环 ──────────────────────────────┐
│  取出当前待执行的节点                   │
│    ↓                                 │
│  执行节点                             │
│    ├── Agent 任务 → 调用 Agent 执行   │
│    ├── 人工审核 → 暂停等待           │
│    ├── 条件判断 → 评估表达式          │
│    ├── Shell 命令 → 执行命令          │
│    └── 通知 → 发送通知               │
│    ↓                                 │
│  更新节点状态和上下文                   │
│    ↓                                 │
│  根据连线确定下一批节点                 │
│    ↓                                 │
│  是否还有待执行节点？                   │
│    ├── 是 → 继续循环                  │
│    └── 否 → 工作流完成                │
└──────────────────────────────────────┘
```

### 6.2 状态管理

| 实例状态 | 说明 |
|----------|------|
| running | 正在执行中 |
| paused | 暂停（等待人工审核） |
| completed | 全部节点执行完成 |
| failed | 某节点执行失败且无法恢复 |
| cancelled | 用户取消 |

| 节点状态 | 说明 |
|----------|------|
| pending | 尚未执行 |
| running | 正在执行 |
| success | 执行成功 |
| failed | 执行失败 |
| skipped | 被跳过（条件分支未走到） |
| waiting | 等待人工介入 |

### 6.3 执行持久化

工作流执行的状态需要持久化到 SQLite，以支持：
- 应用重启后恢复执行
- 人工审核暂停后可恢复
- 历史执行查询

## 7. 工作流模板

### 7.1 内置模板

#### Feature 开发流水线

```
[分析需求] → [编写代码] → [人工审核] → [运行测试]
                                           ↓
                                    ┌──────┴──────┐
                                    ↓             ↓
                              [测试通过]    [测试失败]
                                    ↓             ↓
                              [创建PR]     [修复Bug]
                                    ↓             ↓
                              [通知完成]   [回到审核]
```

#### Bug 修复流水线

```
[分析Bug] → [定位问题] → [编写修复] → [编写测试] → [运行测试]
                                                       ↓
                                                 [测试通过?]
                                                  ↓       ↓
                                              [提交修复] [重新修复]
```

#### 代码审查流水线

```
[扫描代码质量] → [检查安全漏洞] → [生成审查报告] → [人工确认] → [自动修复]
```

#### 依赖升级流水线

```
[检查过时依赖] → [逐个升级] → [运行测试] → [生成变更报告] → [人工审核] → [提交]
```

### 7.2 模板参数化

- 模板定义可参数化的变量
- 用户使用模板时填写变量值
- 支持模板继承和覆盖

## 8. 工作流监控

### 8.1 实时监控视图

- 在可视化图上实时标注每个节点的执行状态
- 已完成节点：绿色
- 执行中节点：蓝色（闪烁）
- 失败节点：红色
- 等待中节点：黄色
- 未执行节点：灰色

### 8.2 执行日志聚合

- 每个节点的日志独立查看
- 提供工作流级别的聚合日志
- 展示关键时间节点和里程碑

### 8.3 执行统计

| 指标 | 说明 |
|------|------|
| 总耗时 | 从开始到结束的总时间 |
| 有效耗时 | 减去人工审核等待时间 |
| 节点成功率 | 各节点的通过率 |
| 累计成本 | 所有 Agent 节点的成本总和 |
| 瓶颈分析 | 耗时最长的节点 |

## 9. 错误处理

### 9.1 节点级错误处理

| 策略 | 说明 |
|------|------|
| retry | 重试 N 次 |
| skip | 跳过失败节点继续 |
| fail | 标记工作流失败 |
| fallback | 执行备用节点 |
| human | 暂停等待人工决策 |

### 9.2 工作流级容错

```json
{
  "error_handling": {
    "default_strategy": "human",
    "max_node_retries": 2,
    "notify_on_failure": true,
    "auto_cleanup_on_cancel": true
  }
}
```

### 9.3 回滚支持

- 工作流取消/失败时，可选择回滚所有节点的变更
- 按节点倒序执行回滚（Git 分支清理等）
- 用户可选择保留某些节点的变更
